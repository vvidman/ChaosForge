# Requirements Pipeline

## Overview
The requirements pipeline transforms human-written Use Cases into developer-ready WorkTasks
through two agent-driven phases. The BusinessAnalyst produces URS from UseCases; the Architect
produces SRS from URS and then decomposes SRS items into WorkTasks. Each step is gated by
a `RevisionGate` where the human can accept, edit, or reject the output.

## Key Concepts

### UseCase
Written by the human. The only domain entity whose content is never generated by an agent.

| Field | Type | Notes |
|---|---|---|
| Id | Guid | |
| ProjectId | Guid | |
| Title | string | |
| Description | string | human-authored, plain language |
| Priority | int | influences BA and SM processing order |

### URS (User Requirements Specification)
Generated by the BusinessAnalyst from one or more UseCases. Represents what the system
must do from a user perspective, without technical detail.

| Field | Type | Notes |
|---|---|---|
| Id | Guid | |
| UseCaseId | Guid | |
| Title | string | |
| Description | string | generated by BA agent |
| RevisionState | RevisionState | tracks human review state |
| HumanEditNote | string? | populated on EditAndAccept; injected into downstream prompts |

### SRS (Software Requirements Specification)
Generated by the Architect from URS. Adds technical detail sufficient to derive WorkTasks.

| Field | Type | Notes |
|---|---|---|
| Id | Guid | |
| URSId | Guid | |
| Title | string | |
| TechnicalDescription | string | generated by Architect |
| RevisionState | RevisionState | |
| HumanEditNote | string? | populated on EditAndAccept |

### Pipeline flow
```
UseCase (human)
  → BA agent → URS → AfterBA RevisionGate
  → Architect agent → SRS → AfterArchitect RevisionGate
  → Architect agent → WorkTask[] (one SRS may produce multiple tasks)
```

### ButterflyEffect on EditAndAccept
When the human edits a URS via `EditAndAccept`, the `HumanEditNote` is injected into the
Architect's prompt for the SRS generation step. When a SRS is edited, existing WorkTasks
derived from it may be invalidated and regenerated. This propagation is handled by
`ButterflyService` in the domain.

## Business Rules
- A URS must be linked to exactly one UseCase; a UseCase may produce multiple URS items
- A SRS must be linked to exactly one URS; a URS may produce multiple SRS items
- `HumanEditNote` is required when `RevisionState = EditedAndAccepted`
- WorkTasks may only be created from SRS items whose `RevisionState` is `Accepted` or `EditedAndAccepted`
- Editing a URS after SRS items have been generated triggers ButterflyService — downstream SRS and WorkTasks must be flagged for regeneration
