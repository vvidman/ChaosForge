# Feature Spec — ButterflyService — EditAndAccept Downstream Propagation

<!-- Reference this file in the implementation agent with: implement @docs/specs/28-butterfly-service.md -->

---

## Context

When a human resolves a RevisionGate with `EditAndAccept`, the edited output must propagate
downstream — otherwise the Architect or ScrumMaster would generate outputs based on stale
data. This is the "butterfly effect" the project is named after: a small human edit
cascades into potentially regenerating SRS items, WorkTasks, or sprint assignments.
`ButterflyService` is triggered by the `RevisionGateResolvedHandler` (spec 26) when
`Action == EditAndAccept`. Depends on: spec 26 (handler knows when to call it).

---

## Domain Impact

- New or modified entity: URS (`HumanEditNote` written), SRS (`HumanEditNote` written),
  WorkTask (may be deleted — see decisions)
- New domain event: none
- New interface: `IButterflyService` (Application/Orchestration/)

---

## Architecture Decisions

- `IButterflyService` lives in `Application/Orchestration/IButterflyService.cs`:
  ```csharp
  public interface IButterflyService
  {
      Task PropagateAsync(RevisionGateDto gate, CancellationToken cancellationToken = default);
  }
  ```
- `ButterflyService` implementation lives in `Application/Orchestration/ButterflyService.cs`.
  It is `internal sealed`, injected via DI. It is an Application service, not Infrastructure —
  it orchestrates via MediatR commands and reads via queries, no direct DB access.
- Propagation rules per gate type:

  **AfterBA (EditAndAccept):**
  - The `HumanEditedOutput` replaces the effective URS content for downstream use.
  - `ApplyHumanEditToURSCommand` is sent for each URS that is referenced in the gate output.
    In practice: the gate's `AgentOutput` is the combined URS summary; the `HumanEditedOutput`
    is the human's revision of that summary.
  - Simplification: because URS items are stored individually and the gate output is a
    combined summary, the `HumanEditNote` is written to ALL URS items for the project
    as a single shared edit note. The Architect will see this note when generating SRS.
  - Any existing SRS and WorkTask records are left intact — they will be overwritten if
    the Architect re-runs (which happens automatically since the AfterArchitect gate was
    not yet opened at this point).

  **AfterArchitect (EditAndAccept):**
  - `HumanEditedOutput` contains the revised SRS summary.
  - Write `HumanEditNote` to all SRS items for the project.
  - Delete all existing WorkTasks whose status is `Backlog` (not yet started) by sending
    `DeleteWorkTaskCommand` — these will be regenerated by the Architect re-run.
    **Note**: `DeleteWorkTaskCommand` does not exist yet — it must be created in this spec.
  - WorkTasks in any other status are left intact.

  **AfterScrumMaster (EditAndAccept):**
  - `HumanEditedOutput` contains the revised sprint plan.
  - Unassign all WorkTasks from their sprint: send `AssignWorkTaskToSprintCommand` with
    a new sprint `Guid` generated here (effectively a re-plan). This is the simplest
    approach — the SM will re-run and re-assign.
  - Alternatively: clear sprint assignments by sending a new command. Create
    `ClearWorkTaskSprintCommand` in this spec to set `SprintId = null`.

- `ButterflyService.PropagateAsync` is called from `RevisionGateResolvedHandler` — add
  this call to spec 26's handler for `EditAndAccept` cases.
- If any command fails during propagation, log and continue — do not roll back the gate
  resolution. Partial propagation is acceptable for a hobby tool.

## Implementation Scope — What must be done

- [x] Create `Application/WorkTasks/Commands/DeleteWorkTaskCommand.cs`:
  - Command: `DeleteWorkTaskCommand(Guid WorkTaskId)`
  - Validator: not empty
  - Handler: `GetByIdAsync` → if not found return Failure → `Delete(task)` → `SaveChangesAsync`

- [x] Create `Application/WorkTasks/Commands/ClearWorkTaskSprintCommand.cs`:
  - Command: `ClearWorkTaskSprintCommand(Guid WorkTaskId)`
  - Handler: `GetByIdAsync` → sets `SprintId = null` via new domain method `ClearSprint()`
  - **Domain change required**: add `ClearSprint()` to `WorkTask` entity:
    ```csharp
    public void ClearSprint()
    {
        SprintId = null;
    }
    ```

- [x] Create `Application/Orchestration/IButterflyService.cs`

- [x] Create `Application/Orchestration/ButterflyService.cs` (`internal sealed`):
  - Implement AfterBA propagation (write HumanEditNote to all URS)
  - Implement AfterArchitect propagation (write HumanEditNote to all SRS, delete Backlog tasks)
  - Implement AfterScrumMaster propagation (clear sprint from all assigned tasks)

- [x] Register `IButterflyService` in `Application/DependencyInjection.cs`:
  `services.AddScoped<IButterflyService, ButterflyService>()`

- [x] Update `RevisionGateResolvedHandler` (spec 26) to inject `IButterflyService` and call
  `PropagateAsync` when `Action == EditAndAccept`

- [x] Run `dotnet build` — zero warnings, zero errors

---

## Out of Scope — What must NOT be done

- Do not re-run agents directly from ButterflyService — workers pick up work on the next
  poll cycle automatically
- Do not implement cascading deletion of SRS/URS on Reject — reject only blocks, it does
  not delete

---

## Test Expectations

- Unit tests required for:
  - AfterBA propagation: `ApplyHumanEditToURSCommand` sent for each URS
  - AfterArchitect propagation: `DeleteWorkTaskCommand` sent for Backlog tasks only;
    non-Backlog tasks not deleted
  - AfterScrumMaster propagation: `ClearWorkTaskSprintCommand` sent for all assigned tasks
  - `WorkTask.ClearSprint()`: sets `SprintId` to null
- Edge cases to cover: no URS items exist for AfterBA propagation → no commands sent,
  no exception

---

## Open Questions

- The `ClearSprint()` domain method has no invariant — it is a simple setter. This is
  acceptable because clearing a sprint assignment is always a valid operation (the SM
  will re-assign on next run).
